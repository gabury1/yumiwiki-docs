# TDD (Test-Driven Development)

> "먼저 실패하는 테스트를 짜고, 그 후에 코드를 작성하라"

프로그래밍 시작 전에 테스트 케이스를 먼저 작성하는 **테스트 주도 개발** 방법론이다. 켄트 백(Kent Beck)이 대중화시켰으며, 단순한 테스트 기술이 아니라 **'설계 기술'**에 가깝다.

## TDD의 순환 주기 (Red-Green-Refactor)

1.  🔴 **Red**: 깨지는(실패하는) 테스트를 먼저 짠다. (코드가 없으니 당연히 실패)
2.  🟢 **Green**: 테스트를 통과할 정도의 **최소한의 코드**만 짠다. (지저분해도 상관없다)
3.  🔵 **Refactor**: 테스트는 통과한 상태를 유지하면서, 코드를 깔끔하게 다듬고 중복을 제거한다.

이 과정을 무수히 반복하며 기능을 쌓아 올린다.

## 왜 하는가? (장점)

**심리적 안정감**: 코드 하나 고쳤다고 다른 곳이 터질까 봐 무서워할 필요가 없다. 테스트가 지켜주니까.

**깔끔한 설계**: 테스트를 먼저 짜려면 함수가 작고 명확해야 한다. 강제로 **결합도가 낮고 응집도가 높은** 코드를 짜게 된다.

**문서 역할**: 테스트 코드는 그 자체로 "이 함수는 이렇게 쓰세요"라는 훌륭한 설명서가 된다.

## 단점 및 오해

- **느리다**: 초기 개발 속도가 20~30% 정도 느려진다. 하지만 나중에 버그 잡는 시간을 생각하면 결국 이득이다.
- **테스의 함정**: 테스트를 위해 코드를 억지로 수정하거나, 의미 없는 테스트(1+1=2 확인 등)만 나열하게 될 수 있다.

## 기타

1. **단위 테스트 (Unit Test)**: TDD의 기본 단위. 함수 하나하나를 작게 쪼개서 테스트한다. [[Java]]라면 **JUnit**, [[JavaScript]]라면 **Jest**를 주로 쓴다.
2. **Coverage**: 내 코드의 몇 %가 테스트되었는지 나타내는 지표다. 100%를 고집하기보다 중요한 로직 위주로 실속 있게 짜는 게 중요하다.
3. **BDD (Behavior Driven Development)**: TDD에서 한 단계 진화하여, 사용자의 "행후" 시나리오를 바탕으로 테스트를 짜는 방식이다.
