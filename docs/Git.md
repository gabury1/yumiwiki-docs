# Git

> 개발자의 타임머신이자 생명줄

리눅스 커널의 창시자 **리누스 토발즈(Linus Torvalds)**가 2005년에 만든 **분산 버전 관리 시스템(DVCS)**이다. 파일의 변경 이력을 추적하고, 여러 명이 동시에 협업할 수 있게 해준다. 개발자에게는 공기와 같은 존재.

## 특징

**분산형**: 중앙 서버(SVN 등)가 터져도 로컬 저장소(Repository)에 모든 이력이 남아있어 복구 가능하다.

**스냅샷 방식**: 변경사항(Diff)이 아닌 파일 시스템의 스냅샷을 저장한다. 속도가 매우 빠르다.

**브랜치 & 병합**: 가지치기(Branching)와 합치기(Merging)가 매우 가볍고 강력하다. 기능별로 브랜치를 따서 개발하고 합치는 워크플로우([[Git Flow]], GitHub Flow)가 일반적이다.

## 핵심 개념

- **Commit**: 의미 있는 변경의 단위(스냅샷).
- **Branch**: 독립적인 작업 공간.
- **Merge**: 브랜치를 합치는 작업.
- **Pull Request (PR)**: [[GitHub]] 등에서 내 코드를 합쳐달라고 요청하는 것. 코드 리뷰의 장.
- **Conflict**: 충돌. 같은 라인을 서로 다르게 수정했을 때 발생.

## 생태계

**호스팅 서비스**: [[GitHub]](압도적 1위), GitLab(자체 호스팅 강자), Bitbucket.

**GUI 툴**: CLI(터미널)가 기본이지만, SourceTree, GitKraken, GitHub Desktop, 그리고 [[VS Code]] 내장 깃 기능도 많이 쓴다.

## 주의점

**강제 푸시(Force Push)**: `git push -f`는 동료의 코드를 날려버릴 수 있는 핵버튼이다. 혼자 쓰는 브랜치 아니면 절대 금지.

**.gitignore**: 보안 키, 빌드 결과물, OS 설정 파일 등은 커밋하지 않도록 미리 설정해야 한다.

## 기타

"리눅스 만들다가 형상 관리 툴이 마음에 안 들어서 2주 만에 뚝딱 만들었다"는 전설이 있다.

처음 배우는 사람에게는 진입 장벽이 좀 있다. `rebase`, `cherry-pick`, `reset` vs `revert` 등 고급 기능은 실수하기 쉽다.
