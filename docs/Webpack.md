# Webpack

> 프론트엔드 자원을 번들링하고 최적화할 때

[[JavaScript]] 모듈 번들러다. 2012년에 나왔고, 프론트엔드 빌드 도구의 표준이었다. 여러 파일을 하나(또는 몇 개)로 묶어준다.

JS, CSS, 이미지, 폰트 등 모든 자원을 모듈로 취급한다. 로더로 변환하고, 플러그인으로 최적화한다.

## 성능

프로덕션 빌드에 최적화 기능 많음. 트리쉐이킹, 코드 스플리팅, 압축. 하지만 설정 복잡하고 빌드 느림.

개발 서버는 [[Vite]]보다 느림. HMR도 프로젝트 커지면 느려짐.

## 생태계

플러그인, 로더가 엄청 많다. [[babel-loader]], [[css-loader]], [[sass-loader]] 등. 뭐든 가능하지만 설정이 복잡.

[[CRA]], [[Next.js]] 등이 내부에서 Webpack 사용(했)음.

## 주의점

**설정 복잡**: `webpack.config.js`가 수백 줄. 진입 장벽 높음. 요즘은 [[Vite]] 추천.

**빌드 속도**: 대규모 프로젝트에서 느림. 캐싱, 병렬 처리 설정 필요.

**레거시 프로젝트**: 기존 프로젝트에서 많이 씀. 유지보수 시 알아야 함.

## 주요 기능

- 모듈 번들링
- 로더 (변환)
- 플러그인 (최적화)
- 코드 스플리팅
- 트리쉐이킹
- Dev Server, HMR

## 기타

Tobias Koppers가 2012년 만들었다. 독일 개발자다. 원래는 개인 프로젝트였는데, [[React]] 생태계에서 채택되면서 급성장했다. [[Create React App]], [[Next.js]], [[Vue CLI]] 모두 내부적으로 Webpack을 썼다.

Webpack 설정 파일이 수백 줄 넘어가는 건 흔한 일이다. "Webpack Configuration Hell"이라는 말이 있을 정도. 로더, 플러그인, 최적화 옵션이 너무 많아서 진입 장벽이 높다. 그래서 요즘은 [[Vite]] 같은 Zero-Config 도구가 인기를 얻고 있다.

Webpack 5(2020년)에서 많이 개선됐다. Module Federation으로 마이크로프론트엔드 지원, 캐싱 최적화, Tree Shaking 개선 등. 하지만 이미 [[Vite]]에 밀리는 분위기였다. 개발 서버 속도에서 Vite를 따라잡기 어렵다.

"Webpack은 죽지 않는다"는 말도 있다. 기존 프로젝트가 워낙 많아서 당분간은 계속 쓰일 거라는 뜻. 레거시 유지보수하려면 Webpack을 알아야 한다. 새 프로젝트는 Vite 추천하지만, Webpack도 여전히 현역이다.
