# HTTP

> 웹에서 데이터를 주고받는 방법을 정의한 프로토콜

클라이언트(브라우저)와 서버가 통신할 때 따라야 하는 규칙이다. 1991년 팀 버너스리가 만들었고, 월드 와이드 웹의 기반이다.

## 탄생 배경

인터넷 초창기, 문서를 공유하려면 통일된 방법이 필요했다. "요청하면 응답한다"는 단순한 규칙을 정해서 어떤 브라우저, 어떤 서버든 통신할 수 있게 됐다.

## 규정 내용

**요청(Request)**:
- 메서드: GET(조회), POST(생성), PUT(수정), DELETE(삭제)
- URL: 자원의 위치
- 헤더: 메타데이터 (Content-Type, Authorization 등)
- 본문: 전송할 데이터

**응답(Response)**:
- 상태 코드: 200(성공), 404(없음), 500(서버 오류)
- 헤더: 메타데이터
- 본문: 응답 데이터

**특징**: 무상태(Stateless). 각 요청은 독립적. 이전 요청 기억 안 함.

## 용도

웹사이트 접속, [[REST]] API 호출, 파일 다운로드. 브라우저, 모바일 앱, 서버 간 통신 모두 HTTP 사용.

## 버전

- HTTP/1.1 (1997) - 연결 재사용
- HTTP/2 (2015) - 멀티플렉싱, 헤더 압축
- HTTP/3 (2022) - UDP 기반([[QUIC]]), 더 빠름

## 생태계

- [[Nginx]], [[Apache]] - 웹 서버
- [[curl]], [[Postman]] - HTTP 클라이언트
- [[HTTPS]] - HTTP + 암호화

## 기타

팀 버너스리가 1991년 CERN에서 월드 와이드 웹과 함께 HTTP를 만들었다. 처음엔 정말 단순한 프로토콜이었다. 요청 한 번, 응답 한 번, 연결 끊기가 전부였다. 지금 보면 비효율적이지만, 그 단순함 덕분에 빠르게 확산될 수 있었다.

HTTP/1.1이 1997년에 나왔는데, 거의 20년 동안 표준이었다. 그만큼 잘 설계됐다는 뜻이다. 하지만 웹사이트가 점점 무거워지면서(이미지, CSS, JS 파일이 수십~수백 개) 성능 문제가 심각해졌고, 결국 2015년 HTTP/2가 나왔다.

"404 Not Found"는 인터넷 문화의 일부가 됐다. 원래는 단순한 에러 코드였는데, 이제는 "없다", "찾을 수 없다"의 은유로 쓰인다. "418 I'm a teapot" 같은 농담 상태 코드도 있다. 만우절 RFC로 만들어진 건데 실제로 구현하는 서버도 있다.

HTTP/3는 TCP 대신 UDP를 쓴다([[QUIC]] 프로토콜). "UDP는 신뢰성 없는데 괜찮아?"라고 생각할 수 있는데, QUIC이 UDP 위에서 신뢰성을 구현한다. 핸드셰이크 횟수를 줄여서 연결 시작이 빠르다. 구글이 주도했고, 크롬과 유튜브에서 먼저 쓰기 시작했다.
