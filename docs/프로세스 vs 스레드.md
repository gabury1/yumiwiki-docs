# 프로세스 vs 스레드

> "공장 건물(Process) 전체를 빌릴 것인가, 그 안의 작업자(Thread)를 늘릴 것인가"

컴퓨터에서 프로그램이 실행되는 기본 단위다. 면접 단골 질문이기도 하지만, [[비동기]] 프로그래밍이나 성능 최적화를 이해하려면 반드시 넘어야 할 산이다.

## 1. [[프로세스]] (Process)

**정의**: 실행 중인 프로그램의 인스턴스. 운영체제로부터 독립된 메모리 영역([[메모리]])을 할당받는다.
- **특징**: "나만의 독방". 다른 프로세스의 메모리에 함부로 접근할 수 없다. 
- **예시**: 크롬 브라우저를 하나 띄우면 하나의 프로세스가 생긴다. 크롬이 죽는다고 메모장(다른 프로세스)까지 죽지는 않는다.

## 2. [[스레드]] (Thread)

**정의**: 프로세스 내에서 실행되는 흐름의 단위. 프로세스가 할당받은 자원을 공유한다.
- **특징**: "한 지붕 여러 가족". 같은 공장 내에서 도구(코드, 데이터)를 공유하며 일한다.
- **예시**: 크롬 브라우저(프로세스) 안에서 탭마다 페이지를 그리고, 파일을 다운로드하고, 영상을 재생하는 일꾼들이 각각의 스레드다.

## 핵심 차이점

| 구분 | 프로세스 (Process) | 스레드 (Thread) |
| :--- | :--- | :--- |
| **자원 공유** | 독립적 (공유 안 함) | 공유함 (Stack만 따로 씀) |
| **생성 비용** | 무겁고 느림 | 가볍고 빠름 |
| **안정성** | 하나 죽어도 남들은 멀쩡함 | 하나가 사고 치면(예외 발생) 프로세스 전체가 죽음 |
| **통신** | 복잡함 (IPC 필요) | 쉬움 (공유 메모리 사용) |

## 왜 스레드를 쓰나?

하나의 공장(프로세스)을 새로 짓는 것보다, 기존 공장 안에서 일꾼(스레드)을 더 뽑는 게 훨씬 효율적이기 때문이다. 하지만 일꾼들이 도구를 공유하다 보니 서로 싸우는 일(**Race Condition**)이 생길 수 있어 주의가 필요하다.

## 기타

1. **컨텍스트 스위칭(Context Switching)**: CPU가 이 일꾼 저 일꾼 번갈아 가며 일을 시키는 과정이다. 이게 너무 잦으면 배보다 배꼽이 더 커져서 성능이 떨어진다.
2. **멀티 프로세스의 현대적 활용**: [[Node.js]]는 기본적으로 싱글 스레드지만, 멀티 코어를 쓰기 위해 `Cluster` 모듈로 프로세스를 여러 개 띄우기도 한다.
3. **자바의 변화**: [[Java]] 21부터는 아주 가벼운 일꾼인 **가상 스레드(Virtual Thread)**가 도입되어 스레드를 수만 개씩 써도 부담 없는 시대가 왔다.
