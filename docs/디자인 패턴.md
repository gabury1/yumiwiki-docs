# 디자인 패턴 (Design Patterns)

> "선배 개발자들이 겪은 삽질의 정수를 모아둔 정량 문서"

소프트웨어 설계에서 자주 발생하는 문제들에 대한 **공통적인 해결책**이다. 효율적이고 유지보수가 쉬운 코드를 짜기 위한 일종의 '비기'이며, 개발자들 간의 '약속된 언어'이기도 하다.

## 왜 필요한가? (건축 비유)

건물을 지을 때 "창문은 이렇게 내고, 환풍구는 저렇게 낸다"는 이미 검증된 방식이 있다. 프로그래밍도 마찬가지다.
- **가독성**: "여기는 싱글톤 패턴 썼어요"라고 한마디만 하면, 다른 개발자가 코드 구조를 즉시 이해한다.
- **재사용성 & 확장성**: 요구사항이 바뀌어도 코드가 쉽게 깨지지 않게 한다.

## 대표적인 패턴들 (GoF의 23개 중 필수)

### 1. 싱글톤 패턴 (Singleton)
- **개념**: "이 클래스의 객체는 딱 **하나**만 있어야 해."
- **용도**: 설정값(Config), DB 연결 풀, [[캐싱]] 저장소 등.

### 2. 전략 패턴 (Strategy)
- **개념**: "알고리즘을 갈아 끼울 수 있게 만들자."
- **용도**: 결제 수단(카드, 네이버페이, 카카오페이)을 런타임에 바꿀 때.

### 3. 옵저버 패턴 (Observer)
- **개념**: "상태가 바뀌면 나한테 알려준 애들(구독자)한테 다 알려줘."
- **용도**: [[JavaScript]]의 이벤트 리스너, [[Vue]]/[[React]]의 상태 감지.

### 4. 팩토리 패턴 (Factory)
- **개념**: "객체 생성을 직접 하지 말고 공장(Factory)한테 시키자."
- **용도**: 복잡한 조건에 따라 다른 객체를 만들어야 할 때.

## 주의점: 패턴 만능주의 (Anti-Pattern)

패턴은 도구일 뿐이다. 간단한 코드에 억지로 복잡한 패턴을 적용하는 것을 **Over-engineering**이라고 한다. "코드를 패턴에 맞추지 말고, 문제 해결을 위해 패턴을 써야 한다."

## 기타

1. **GoF (Gang of Four)**: 디자인 패턴 하면 꼭 나오는 용어다. 1994년에 "Design Patterns"라는 책을 쓴 4명의 저자를 '4인조 대장'이라 부르며 패턴의 시초가 되었다.
2. **헤드 퍼스트**: 디자인 패턴이 너무 딱딱하다면 **"Head First Design Patterns"**라는 책을 강력 추천한다. 그림과 퀴즈로 아주 재밌게 설명해준다.
3. **안티 패턴**: 하면 안 되는 설계 습관도 패턴으로 정리되어 있다. 예를 들어 모든 로직을 다 때려넣은 하마 같은 클래스는 **God Object**라고 불린다.
