# 롤링 업데이트 (Rolling Update)

> 한 대씩 차례대로 바꾸는 점진적 교체

**롤링 업데이트**는 실행 중인 인스턴스를 하나씩 순차적으로 새로운 버전으로 교체하여 가동 중단 없이 전체 시스템을 업데이트하는 전략이다.

## 작동 방식

1. **시작**: 총 3대의 서버가 구버전(V1)을 돌리고 있다.
2. **교체 1**: 1번 서버의 연결을 끊고 신버전(V2)으로 업데이트한 뒤 다시 연결한다. (V2:1, V1:2)
3. **교체 2**: 2번 서버를 같은 방식으로 업데이트한다. (V2:2, V1:1)
4. **완료**: 마지막 3번 서버까지 교체하면 모든 서버가 신버전(V2)이 된다.

## 장점

- **가동 중단 최소화**: 일부 서버가 항상 서비스 중이므로 무중단 배포가 가능하다.
- **리소스 효율성**: 블루-그린 배포처럼 전체 환경을 두 배로 준비할 필요가 없어 비용 면에서 유리하다.
- **간편한 자동화**: 대부분의 오케스트레이션 도구([[Kubernetes]], ECS 등)가 기본적으로 지원한다.

## 단점

- **호환성 이슈**: 업데이트 중간에 구버전과 신버전이 공존하므로, 사용자가 두 버전을 번갈아 마주칠 수 있다. (API 호환성, DB 스키마 주의 필요)
- **느린 배포 속도**: 서버 한 대씩(혹은 일정 비율씩) 순차적으로 진행되므로 전체 배포 완료까지 시간이 걸린다.
- **롤백의 어려움**: 중간에 문제가 터지면 이미 업데이트된 인스턴스를 다시 하나씩 구버전으로 되돌려야 한다.
