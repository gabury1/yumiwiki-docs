## 비교: 나무위키 vs YumiWiki

### 나무위키의 강점
1. **구체적 수치**: 성능 벤치마크, 시장 점유율, 버전별 변화
2. **비판적 균형**: 장점만이 아닌 단점도 솔직하게
3. **각주와 참고 자료**: 출처 명확
4. **논쟁 포함**: 커뮤니티 의견 차이 반영

### YumiWiki의 강점
1. **실무 중심**: 개발자가 실제로 겪는 문제 위주
2. **간결함**: 나무위키보다 읽기 쉬움
3. **링크 기반**: 개념 간 연결 용이
4. **일관된 톤**: 편안하고 친근한 말투

## 구체적 개선 제안

### 보안.md 개선안
```markdown
## 주요 위협

### SQL Injection
공격자가 SQL 쿼리를 조작해서 DB에 접근. `' OR 1=1--` 같은 입력으로 인증 우회.
방어: Prepared Statement, ORM, 입력 검증.

### XSS (Cross-Site Scripting)
악성 스크립트를 웹페이지에 삽입. `<script>alert('XSS')</script>`
방어: 출력 이스케이프, CSP 헤더, HttpOnly 쿠키.

### CSRF (Cross-Site Request Forgery)
사용자 권한으로 의도하지 않은 요청 실행.
방어: CSRF 토큰, SameSite 쿠키.
```

### 캐싱.md 개선안
```markdown
## 캐싱 전략

### Cache-Aside (Lazy Loading)
애플리케이션이 캐시를 직접 관리.
1. 캐시 확인 2. Miss면 DB 조회 3. 캐시에 저장
장점: 필요한 것만 캐싱. 단점: 첫 요청 느림.

### Write-Through
쓰기 시 캐시와 DB 동시 갱신.
장점: 일관성 높음. 단점: 쓰기 느림.

### Write-Behind
쓰기를 캐시에만 하고 비동기로 DB 반영.
장점: 쓰기 빠름. 단점: 장애 시 데이터 유실 가능.
```

### 확장성.md 개선안
```markdown
## 확장성 수준

- **10명 동시 접속**: 기본 설정으로 충분
- **100명**: DB 인덱스, 캐싱 필요
- **1,000명**: 로드 밸런서, DB 리플리케이션
- **10,000명**: CDN, 캐시 클러스터, DB 샤딩
- **100,000명**: 마이크로서비스, 메시지 큐, 글로벌 분산

실제 사례:
- 쿠팡: 수백만 동시 접속 대비 MSA 아키텍처
- 카카오톡: 샤딩으로 5천만 사용자 처리
```

## 결론

현재 YumiWiki는 **Implementation 문서는 우수**하지만, **Concept/Mechanism은 개선 여지**가 큽니다.

나무위키처럼 "백과사전"을 만들기보다는, **실무 개발자를 위한 레퍼런스**라는 정체성을 유지하면서 **깊이만 보강**하면 훌륭한 문서 세트가 될 것입니다.

특히 Spring Boot, Django, MySQL 같은 문서는 이미 나무위키 수준의 품질을 보여주고 있습니다. 이 수준을 다른 문서에도 적용하면 좋겠습니다.
